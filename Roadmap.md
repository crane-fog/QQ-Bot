# QQ-Bot 文档与可用性 Roadmap

## 背景

这份 Roadmap 的目标是先明确：

- 项目目前为什么不够开箱即用
- 需要补哪些文档
- 文档之外，是否还有少量配套改动值得一起做
- 这些工作应该按什么顺序推进

[建议推进顺序](#建议推进顺序)

## 当前问题归纳

### 运行依赖没有分层说明

项目启动不仅需要 Python 和依赖安装，还依赖：

- OneBot HTTP 监听端
- 部分插件额外依赖 PostgreSQL、Redis、LLM Key、语音服务等

现在这些条件分散在代码和模板里，新接手的人很难判断需要哪些才能进行最小成功启动。

### 缺少初始化配置

数据库没有建表脚本，需要手动创建，且需要从代码中提取表结构。

额外的，LLBot 也需要自行编写运行脚本，用于 systemd 管理。

### 插件依赖关系不透明

插件系统是这个项目的核心，但目前缺少一份可检索的插件清单。实际使用时会遇到这些问题：

- 插件触发词只能看代码
- 插件需要哪些配置项只能看代码
- 插件是否依赖数据库/Redis/外部 API 只能看代码

## 建议推进顺序

- 补充插件文档
- 补充部分急需的数据库模型的建表脚本
- **Gitea Webhook 功能**
- 统一数据库模型
- 重写 WebController，目前弃用
- config toml
- 为 Api 引入 TypedDict
- 补充 LLBot systemd 服务脚本

## 未来探索

- 由于 LLBot 使用 Hook 端 QQ，需要启动无头 QQ，资源占用大，可以考虑换成 Napcat 监听端，以大量减少资源占用。
- 实现插件热重载功能
